# 04/03炉资源必要资源量与工作分割

炉资源可以放100个中间品，则必要资源量设为0.01（做一个产品占用炉的资源量的0.01）。假设炉资源的必要资源量为0.01，当有3个订单，每个订单的制造数量为50，且这三个订单加工条件相同，则需要将3个订单进行工作分割成150份，然后由于炉资源的必要资源量为0.01，即一次可以处理2个订单的100份工作，所以一次性可以排两个订单进去；若不进行工作分割，则炉资源会按订单计算，以为此时有3/100个任务，于是就将3个订单一次性排进一趟炉资源处理中，而这是不对的。若觉得订单分割较为麻烦，可在资源表中对炉资源的`分派资源量标志`设置为`与制造数量成比例`，此时炉资源就会自动识别订单中的制造数量，以制造数量为单位进行分派，而不是以订单数量为单位进行分派。

# 04/03静态分割与动态分割

- 静态分割

工作表中，系统必须先正常展开一次，把工序、物料、中间品都算出来，只有生成了工作，才可以编辑分割参数，分割参数编辑好后，要想使其生效，必须再展开一次，使系统按照编辑好的分割参数进行排程。（显示设置-最上位收集条件：分割源工作以外【只显示实际工作，不显示工作根】）。

品目表中对最终品目或中间品进行分割，不需要二次展开，但是对中间品的分割要注意，只有展开一次后才会显示中间品，此时才能对中间品进行分割。

- 动态分割

资源表中，即便对机器设置了资源分割数量，排程没有用到该机器时，也不会生成相应的分割结构，只有这台机器真正被排程用到时，系统才会生成对应的分割结构，所以是动态分割。制造BOM表也同理，只有这个品目 / 工序真正被使用时，才会生成对应的分割结构。

计划参数表中的允许分割属性在展开和排产时生效，此时即便表中设置了分割，优先级也是取决于计划参数表，同时还要注意计划参数表的分割工作是否靠拢在一起。在工艺bom中只显示使用指令，在物料bom中只显示输入指令。

在订单展开时订单下面挂的是工作，工作上挂的输入输出指令（未排程，资源不确定是否使用，故没有使用指令），排产后发生变化，工作下面挂任务（前设置/制造/后设置），任务下面挂输入输出指令。原料数量在输入指令中的前设置/制造/后设置都可以填，填在哪个参数下，排程时指令就挂在哪个任务下。

# 04/10BOM分类

工艺BOM中的字段通常要加在使用指令模板中，物料BOM的字段通常加在输入指令模板中，而均有的字段则通常加在所有指令模板中。这是由于某些与指令相关的字段在从其他字段取值时只能在该指令模板所拥有的字段中取到，例如，制造字段在输入、输出、使用指令模板中均有，但是如果制造字段属于输入指令模板，并且要从其他字段取值时，只能从输入指令模板中拥有的字段中取值，而无法从其他指令模板中取到。

## 工艺BOM分类

![image-20260410145803339](https://img.tynote.cn/img/typora/20260410145812567.png#800w)

### 标准工艺BOM

定义：产品分类维度的通用工艺模板，不绑定具体物料，只定义<font color='blue'>某类产品</font>的标准工艺规则。

作用：为料号级 BOM 提供通用工艺模板，避免重复维护。

标准工艺BOM两种生成方式：①按照制造BOM表结构手工填入②手工维护分表，最终将分表拼接成标准工艺BOM

分表有6张：工艺BOM-工艺路线、工艺BOM-与主资源对应、工艺BOM-与副资源对应、工艺BOM-主资源与副资源对应、工艺BOM-工时、工艺BOM-复杂工艺。

注：维护工艺BOM-工时表时，若同一工序的多个主资源所需工时相同，也就是制造字段的值相同，换句话说，也就是工序所需工时与资源无关，那么此时可以省略主资源字段的值，将同一工序的多条工时相同的记录合并为一条记录，提高该分表的维护效率。

### 料号级工艺BOM

定义：绑定具体物料（料号）的**单品工艺标准**，是连接标准与订单的核心中间层。

生成方式：①从标准类型工艺 BOM 生成（可直接复用通用模板，快速生成单品工艺）②手工维护分表，最终将分表拼接成料号级工艺BOM③直接手动输入

作用：为订单级 BOM 提供**单品工艺依据**

### 订单级工艺BOM

定义: 一定是当前时间内确定的要进行生产的订单。绑定具体生产订单的**个性化工艺**，是 APS 排程、车间现场执行的最终数据。

注：如果是未来一段时间内未开出制造订单的，我们称为计划订单。

生成方式：①从料号级工艺 BOM 生成（可直接复用通用模板，快速生成单品工艺）②直接手动输入

### 标准工艺BOM与订单工艺BOM的制造字段的取值区别

在HANSAFLEX项目中，有四个BOM表（实际上是一张底表，按照不同的业务场景类型筛选并显示不同的数据而已）。

不同BOM表中的制造字段在排产时的区别：

在标准工艺BOM中，制造字段的意思通常就是单位需求量，即该工艺路线生产一个最终品目所需要的原料或者某个中间品目的数量，所以在导入客户订单分派工作时，实际的原料或中间品的需求量为`制造字段的值×订单数量`

但在其订单工艺BOM表中，现有两个字段分别是需求数量和已领数量，所以实际需要生产的数量是会发生变化的，实际需要生产的数量=需求数量-已领数量。订单工艺BOM本质上是BOM表，但由于一些字段的原因，在用户视角更具有订单属性，同时由于1️⃣`需求数量字段-已领数量=未领数量`已经给出了该道工序所需要的原料或中间品目的数量，2️⃣在BOM中计算实际的原料或中间品目的需求量时会拿制造字段的值×订单数量。而我们此时仅靠未领数量就可以计算得出，不再需要制造字段的值去乘订单数量，所以需要在其乘之前将其抵消，所以制造字段的值应该修改为`（需求数量-已领数量）/订单数量`，也就是`未领数量/订单数量`，这样程序在计算实际需要生产的数量时的步骤是`制造字段的值×订单数量=（需求数量-已领数量）/订单数量×订单数量=需求数量-已领数量=未领数量`

   任务：研究HANSAFLEX模型中工作表的字段[不足数量]和订单表字段[是否齐套]的表达式怎么写

   在之前04/03章节中已经学过，工作输入指令In挂在工作上的三种情况：①常规订单展开时挂在工作下②常规计划参数，也就是排产时挂在工作的任务（前设置、制造、后设置）下，原料填在哪个任务下就挂在哪个任务下，通常填在`制造`下③常规订单展开和常规计划1参数均使用时，工作输入指令所挂位置取决于二者执行的先后顺序

   ![情况①](https://img.tynote.cn/img/typora/20260427010649430.png#800w)

   ![情况②](https://img.tynote.cn/img/typora/20260427010807210.png)

   目的：要在订单表中显示出当前订单订单是否齐套，即所需物料是否齐全

   原料的不足数量体现在工作输入指令表中，通常为`In`，如下图所示

   ![工作输入指令表](https://img.tynote.cn/img/typora/20260427010236110.png#800w)

   要想在订单表中显示[是否齐套]，不必须要依靠工作输入指令表中的`不足数量`字段，而工作输入指令表与订单表的关系是工作输入指令表->工作表->订单表，所以订单表需要使用工作表，而工作表需要使用工作输入指令表

   我的做法：将工作输入指令表的`不足数量`传递到工作表上，所以情况①时在工作表中新建字段[不足数量1]，使用maxif()函数取到工作输入指令中的最大值，有一个原料不足数量>0，则将来该原料所在订单一定不齐套，取值表达式为MaxIF(ME.工作输入指令,TARGET.不足数量>=0,TARGET.不足数量)，情况②时新建字段[不足数量2]，取值表达式为MaxIF(ME.子对象,TARGET.WorkUser_RQ>=0,TARGET.WorkUser_RQ)。（因为情况②时使用情况①的表达式取值会取到当前工作所在的子对象下面，无法反映到工作上来，如下图，所以将来也无法反映到订单中，故需更换表达式）

   ![image-20260427012235407](https://img.tynote.cn/img/typora/20260427012235767.png#800w)

   然后订单表中新建字段[是否齐套]，表达式为If(Max(ME.Operations,TARGET.RQ2)>0||Max(ME.Operations,TARGET.RQ)>0,'不齐','齐')，意为只要工作表中的字段[不足数量1]和[不足数量2]有一个不为0则为不齐套，反之为齐套。

   董老师的指导的正确做法：不可使用子对象，如果将来要取得值是字符串，则可能报错，且工作表中使用了两个字段来表达最终的不足数量，不太合适，尽量使用一个表达式和一个字段。我们观察到工作在订单展开和排产下的区别在于是否未分派，情况①订单展开时工作输入指令挂在工作下，为未分派状态，直接取当前工作的工作输入指令的不足数量的最大值即可，情况②排产时挂在工作的制造任务下，为非未分派状态，所以要取工作的制造任务的工作输入指令的不足数量，即If(ME.分派标志=='未分派',MaxIF(ME.工作输入指令,TARGET.不足数量>=0,TARGET.不足数量),Max(ME.制造任务,TARGET.工作输入指令.不足数量))。

   最终订单表中新建字段[是否齐套]，表达式为If(Max(ME.工作,TARGET.'[不足数量]')>0,'不齐','齐')。

二、研究董老师在hansaflex数据库中写的视图分别实现了什么功能，并且为什么要写视图

观察hansaflex模型的导入/导出表可发现，导入订单物料BOM、工作单、库存、采购单、订单工艺BOM时采用的是视图

【v_Inv】:库存视图，使用SUM(Qty) AS Qty,    DATEADD(DAY,-7,GETDATE()) AS LET用于汇总库存并设置交期，交期为当前时间的七天前。'Inv'+'-'+Branch+'|'+LocationID+'|'+Item AS Code拼成库存记录的“唯一编码” Code

库存交期设置为7天前的原因：①库存默认为现在已经在仓库了②它的可用时间应该早于当前排程时间③设置为7天只是通常的保险做法。综上所述，不是说这批库存真的是 7 天前到的，而是说，对 APS 来说，这批库存在排程开始前就已经可用了。

有库存表但采用库存视图的原因：原表结构不一定适合 APS 直接读取，APS 更需要统一格式，以及拼接字段、添加字段、补充某些功能等。

【v_MO】:工作单视图，用 Code + '|' + RowNum 生成唯一订单行编码,计算实际需求量Qty = OrigQty - ReceiptQty

【v_ORMBom】:订单物料表视图，把订单号和行号拼成一个最终件标识a.Code+'|'+CAST(a.RowNum AS VARCHAR(10)) AS FinalItem，'In'+CAST(FORMAT(ROW_NUMBER() OVER(PARTITION BY a.Code,a.RowNum ORDER BY a.Component),'000') AS VARCHAR(10)) AS InsCode为每个订单行里的物料自动生成顺序号，ISNULL(a.IssuedQty,0) AS IssuedQty物料需求量为空则置0，避免为空在Asprova中报错。

从订单工艺 BOM 里，找出每个订单行的“第一道工序”，再把订单物料挂上去。

```
LEFT JOIN 
(
    SELECT DISTINCT t1.Code,t1.RowNum,t1.ProcNo,t1.ProcCode
    FROM t_OrPBom t1
    INNER JOIN (
        SELECT DISTINCT Code,RowNum,MIN(ProcNo) AS ProcNo
        FROM t_OrPBom
        GROUP BY Code,RowNum
    ) t2 
    ON t1.Code=t2.Code 
    AND t1.RowNum=t2.RowNum 
    AND t1.ProcNo=t2.ProcNo
) b 
ON a.Code=b.Code AND a.RowNum=b.RowNum
```
